5 # AURA: Another Universal RPC, Actually
6 
7 Aura is a simple and fast universal RPC library that allows you to quickly interface
8 with your favourite hardware without reinventing the wheel. Unlike most of the RPC
9 libraries, it is designed to link you to simple embedded devices and take care of their
10 weirdness, namely:
11 
12 * It does all the job serializing and deserializing data, taking all the care of foreign endianness
13 
14 * It doesn't use any complex data formats (e.g. XML) that some 8-bit avr will have a hard time
15  parsing.
16 
17 * Remote 'nodes' can go offline and online multiple times and it's not an error (unless you want it to be)
18 
19 * Remote 'nodes' can even go offline and come back online with a different set of exported functions. And it
20  still will not be an error, unless you want it to be.
21 
22 * Aura provides hooks to override memory allocation, allowing you to interface with DSPs that require
23  a special memory allocation technique for performance reasons, e.g. ION Memory Manager. And it allows
24  you to avoid needless copying of serialized data. e.g. zero-copy
25 
26 * Provides both sync and async API
27 
28 * Written in pure C with as little dependencies as possible
29 
30 * Has lua bindings (Want your favourite language supported as well - roll up the sleeves and contribute!)
31 
32 * OpenSource
33 
34 # TL; DR (In C, Sync API)
35 
36 ```C
37 int main() {
38  int ret;
39  struct aura_node *n = aura_open("dummy", NULL);
40  aura_wait_status(n, AURA_STATUS_ONLINE); // Beware of TOCTOU! See docs!
41  struct aura_buffer *retbuf;
42 
43  ret = aura_call(n, "echo_u16", &retbuf, 0x0102); //echo_u16 is a method on a remote device
44  if (ret == 0) {
45  uint16_t rt = aura_buffer_get_u16(retbuf);
46  aura_buffer_release(n, retbuf);
47  printf("Call result: %d\n", rt)
48  }
49  aura_close(n);
50  return 0;
51 }
52 ```

84 # Terminology
85 
86 Aura runs on a 'host machine'. E.g. Your laptop, Raspberry PI, OpenWRT router. Anything
87 with a recent linux ditro would do just fine.
88 
89 Aura connects to different devices called 'nodes' via some transport. As you might've guessed this
90 needs a transport plugin. A full list of available plugins is somewhere below in this file.
91 
92 When going online a node reports (in a transport-specific way) about stuff it has. It is called an
93 and aura fills in an 'export table'. Export table is just a list of objects that a node 'exports'.
94 
95 Each object can be either a 'method' or 'event'. It is identified by it's name and a numeric id that is
96 used internally to issue a call.
97 
98 A 'method' is just your good old RPC call, like:
99 
100 result1, result2 = node.mymethod(arg1, arg2);
101 
102 Host issues all method calls.
103 
104 'Events' on the other hand are issued by a remote node. Normally you would want to use the async API for that!
105 
106 Basically that's all you need to know. More details in the doxygen docs.

132 # Available transports:
133 
134 * dummy - A dummy transport for unit tests. We love unit-tests. Nuff said!
135 
136 * sysfs-gpio - A 'transport' that allows you to control host PC GPIO lines using the sysfs interface. Should serve as a more complex transport plugin example.
137 
138 * susb - A simple usb transport that does control transfers to a usb device according to a config file. Using this one you can interface with simplest possible USB devices that have no flash for any other logic. Doesn't support events. Needs lua, since the configuration file for it is just a lua script.
139 
140 * usb - A more complex transport over USB. Supports events and methods, reads the information from the target device
141 
142 * nmc - NeuroMatrix DSP transport. Requires experimental Easynmc Framework and ION Memory Manager. Aura must be running on a SoC equipped with a NeuroMatrix DSP, e.g. K1879ХБ1Я.
143 
144 Want more? Contribute!

146 ## simpleusb (a.k.a. susb)
147 
148 simpleusb transport maps control transfers to methods. Simple as that. No events whatsoever. All methods require at least two arguments. wValue and wIndex (See USB spec). The transport accepts a path to the configuration file as it's option.
149 
150 A typical configuration file may look like this:
151 
152 ```lua
153 device =
154 {
155  vid = 0x1d50;
156  pid = 0x6032;
157  vendor = "www.ncrmnt.org";
158  product = "aura-susb-test-rig";
159  --serial = "blah";
160  methods = {
161  [0] = NONE("led_ctl"),
162  [1] = NONE("blink"),
163  [2] = READ("read", UINT32, UINT32),
164  [3] = WRITE("write", UINT32, UINT32),
165  };
166 }
167 
168 
169 return device
170 ```
171 
172 The config above describes 4 methods:
173 
174 ```
175 --- Dumping export table ---
176 0. METHOD led_ctl( uint16_t uint16_t ) [out 4 bytes] | [in 0 bytes]
177 1. METHOD blink( uint16_t uint16_t ) [out 4 bytes] | [in 0 bytes]
178 2. METHOD uint32_t uint32_t read( uint16_t uint16_t ) [out 12 bytes] | [in 8 bytes]
179 3. METHOD write( uint16_t uint16_t uint32_t uint32_t ) [out 12 bytes] | [in 0 bytes]
180 -------------8<-------------
181 
182 ```
183 
184 vendor, product and serial strings can be omited. If so the transport will match any.
185 
186 NONE stands for a control packet with no data phase.
187 
188 READ stands for *surprise* reading data from device
189 
190 WRITE stands for writing data to device
191 
192 Your hardware should pack the data without any padding between fields. E.g. Use __attribute__(("packed")) for your structs in firmware.
193 
194 N.B.
195 
196 VUSB-based devices (or libusb?) do not seem to play nicely when we have the setup packet only with no data part.
197 Transfers may fail instantly at random without any data delivered. A typical run at my box gives:
198 
199 9122 succeeded, 878 failed total 10000
200 
201 The distribution of failures is more or less even. See tests-transports/test-susb-stability-none
202 
203 Adding just one byte of data to the packet fix the issue. Posible workarounds here are:
204 
205 1. Retry N times
206 
207 2. Add just one dummy byte for the transfer
208 
209 Since nobody reported this workaround breaking support for their hardware (yet!) - we'll do it the second way.
210 For now.

212 ## usb
213 
214 This is a full-blown aura RPC over the usb bus with events.
215 The transport's option string looks like this:
216 
217 "1d50:6032;vendorstring;productstring;serial"
218 
219 To match *ANY* serial, product, vendor:
220 
221 "1d50:6032;;;"

224 ## nmc
225 
226 You can use this transport to call remote methods running on RC Module's NeuroMatrix DSP core. This transport requires that you run it on a SoC with NeuroMatrix DSP and experimental easynmc driver, e.g. K1879ХБ1Я.
